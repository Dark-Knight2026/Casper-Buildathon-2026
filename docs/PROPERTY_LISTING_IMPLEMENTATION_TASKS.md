## The one fact that changes everything

The `feat/properties` backend is **built**. Per `properties_api.md`, `/api/v1/properties` and `/api/v1/listings`
(+ favorites/applications/viewings re-homed onto `listingId`) are implemented (some parts `[stubbed]` —
KYC always-true, T2 absent, on-chain `null`, EXIF/IPFS faked). So the property-listing block is **no longer
BE-blocked** — the FE work is real integration, not waiting. The matching rows in `FRONTEND_MVP_TASKS.md`
(⛔ BE-BLOCKED) are now stale and should be updated.

The whole block assumes a **single flat `Property`** that conflates the physical asset and the rental offer,
and the data layer talks to **Supabase**, not the backend. The contract requires a **Property (physical, RESO
camelCase)** + **Listing (offer: `intent`/`state`/`terms`/`media`/`provenance`/`onChain`)** split.

## F. On-chain property registration (`PropertyRegistry` — Casper)

> Contract: `PropertyRegistry` (Odra) — `2025_anthony_leasefi/src/property_registry.rs`.
> Schema copied to [`docs/casper_contract_schemas/property_registry_schema.json`](./casper_contract_schemas/property_registry_schema.json).

### Contract shape (what the chain stores)

`PropertyRecord` keyed by a `property_id` (`U256`, auto-incremented):

| Field          | Type             | Notes                                                     |
| -------------- | ---------------- | --------------------------------------------------------- |
| `issuer`       | `U256`           | UserRegistry **user id** of the landlord (not a wallet)   |
| `token`        | `Option<Key>`    | fraction-token address; set later, while `Draft`          |
| `total_supply` | `U256`           | intended ownership-token supply (feeds fractionalization) |
| `metadata_uri` | `String`         | IPFS URI / content hash of the canonical property payload |
| `status`       | `PropertyStatus` | `Draft → Active → {Paused, Sold, Liquidating → Closed}`   |

Write entry points (all mutating, `is_contract_context`):

- `create_property({ issuer, total_supply, metadata_uri }) -> property_id` — creates a `Draft`, emits `PropertyCreated`.
- `set_property_token(property_id, token)` — only while `Draft`; reverse-lookup guarded (one token ↔ one property), emits `PropertyTokenSet`.
- `set_metadata_uri(property_id, metadata_uri)` — only while `Draft`, emits `PropertyMetadataSet`.
- `set_property_status(property_id, status)` — `Active` **requires** a token set; no demote-to-`Draft`; `Sold`/`Closed` terminal; `Liquidating → Closed` only. Emits `PropertyStatusSet`.

Reads: `get_property`, `get_property_token`, `get_property_id_by_token(token) -> Option`, `is_property_active(id) -> bool`.

### Actor model — the landlord signs from the FE

The `PROPERTY_MANAGER` caller-role gate is **being removed from the contract** — `create_property` / `set_*`
become callable by the landlord directly. So on-chain registration is **FE-driven**: the landlord signs each
deploy through **CSPR.click** (the wallet-only signing layer — [[project_leasefi_auth_wallet_decoupling]]) and
the FE submits it to casper-test. The landlord is both **caller** and **`issuer`** (`issuer` = the landlord's
own on-chain user-id, resolved from the indexer — [[project_leasefi_onchain_registration_indexer]]).

This is **signing, not wallet management** — keep it consistent with [[project_leasefi_wallet_invisible_to_user]]:
the CSPR.click sign prompt is expected, but **no "connect wallet" / "create wallet" CTA**. Chain **reads**
still come from the indexer / a CSPR.cloud RPC, not hand-rolled contract queries in the browser.

### Two distinct on-chain surfaces — don't conflate
