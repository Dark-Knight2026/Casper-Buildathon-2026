Phase 1 limitation · PM acting on behalf
Property managers can draft leases on behalf of landlords via PMAuthorization, but cannot sign for them. The landlord must complete the EIP-712 signature themselves on their own wallet. Phase 2 introduces delegated signing once Casper Smart Account integration lands. Implementation: PM completes Steps 1–5 of the wizard, then the wizard transitions to a handoff state at Step 6 ("Awaiting landlord signature") — the landlord receives a notification with a deep link back into the wizard at Step 6.
Signing journey copy rules
GOOD"You signed" / "Waiting for tenant" — neutral, observational
GOOD"Lease is active" — completion of signing, not the lease term itself
GOOD"Save draft" persistent throughout wizard — drafts persist server-side per draft endpoint pattern
NEVER"Your tenant hasn't signed yet" — borderline accusatory framing
NEVER"Sign now or lose it" — manufactured urgency; LeaseFi has no scarcity
NEVER"Lease complete" — implies the lease term itself completed
NEVER"Send a reminder" CTA that nags the tenant directly — landlord can resend the link, the system does not nag
Watch-outs
  • Lease activation is two separate on-chain transactions. LeaseFactory.create_lease (Tx A) deploys the Lease and DepositVault contracts, mints NFTs, collects the 2% lease execution fee, and verifies both EIP-712 signatures. DepositVault.deposit_funds (Tx B) is a separate transaction the tenant submits afterward — no signatures, just funder/amount/currency. Each has independent atomic rollback. The pending-transaction store must track them independently.
  • EIP-712 signatures are function arguments, not stored nonces. Verified inline by create_lease, never written to chain. If Tx A reverts, the same signatures retry cleanly. If Tx A succeeds and Tx B reverts, no signatures are needed for Tx B retry — deposit_funds doesn't require them.
  • Stranded-lease risk · escalate to SC team. Between Tx A success and Tx B success, the tenant has a created lease with an empty vault. If the tenant disappears, the lease is stranded on-chain — contracts exist, execution fee collected, but no deposit on file. Confirm with SC team that a clean cancellation path exists (e.g., Lease.cancel_unfunded after a grace period). Without one, bad-faith tenants can create leases cheaply and block landlords from re-leasing until manual cleanup. Render an explicit "lease created · awaiting deposit" state in the UI; do not hide this transitional period.
  • Save draft is mandatory on every step. Per the draft endpoint pattern. Landlords abandon multi-step forms; the draft is how they come back. Server stores the draft, not the browser.
  • The handoff is asynchronous. Don't lock the landlord's session waiting for the tenant. Landlord may sign and disappear for days; tenant may sign at any point within the 7-day window. The "waiting" state is a steady-state view, not a loading spinner.
  • Landlord cannot modify the draft after signing. EIP-712 signature is over the immutable payload. Edits invalidate the signature. If the landlord needs to change terms, cancel the draft and start over — and message that clearly.
  • Lease becomes active when tenant submits LeaseFactory.create_lease. The tenant is the on-chain submitter, paying gas, supplying both signatures and funding the deposit in a coordinated submission. The contract verifies everything atomically.
  • Server stores sha256(invite_token) only. Even if the link is intercepted via screenshot, referrer leak, or shoulder-surfing, the attacker still needs the tenant's wallet to sign.
  • Resend generates a new token, invalidates the old. If the tenant lost the link or the landlord suspects compromise, resending rotates the token. Old link returns 404.
  • Blank-tenant pattern — verify with SC team. Landlord signs with tenant: ZERO_ADDRESS; first wallet to sign claims the role. This is an open question per CLAUDE.md §12. Don't ship UX dependent on it until confirmed.
  • EIP-712 schema is LeaseAgreement, verifyingContract = LeaseFactory address. Get either wrong and signatures verify cryptographically but reject on-chain. Use @leasefi/types builders only; no inline typed-data construction.
  • FL state cap on security deposit is 2× monthly rent. Validate client-side, contract-side, and at submission. Surface jurisdiction context if other states have different caps in Phase 1+ (TX, TN, VA).
  • Tenant's KYC, fee approval, and lease signature happen in a single session. Don't lose progress between them. If the tenant abandons after approving the fee but before signing, the next session resumes from sign.
  • SHA-256 vs IPFS CID — keep them separate. The on-chain lease_document_hash is SHA-256. The CID is the retrieval address only. Always re-hash retrieved PDFs and compare to the on-chain hash before trusting content. Never trust the CID alone — IPFS gateways can serve different content for the same CID under attack.
  • Notification triggers fire on state transitions, not on polling. State 3→4 notifies tenant. State 4→5 notifies landlord. State 6→7 notifies both. Subscribe to indexer SSE. Per Off-Chain Spec §7.1.1.
  • Resumability is a hard requirement. Landlord opens the wizard, fills 4 steps, closes the tab, returns 3 days later. The wizard must restore state from the server-persisted draft — not from localStorage or sessionStorage. Browser-persisted draft state is forbidden. Test this path in Playwright; it's the #1 regression target.
  • IPFS pin tokens are server-issued, 5-minute TTL. Frontend never holds Pinata or web3.storage API keys. If pin fails after upload, the PDF is on the server but not yet on IPFS — retry without re-uploading.
  • Hash verification is the security boundary. Compute SHA-256 client-side on upload. Send hash + PDF to server. Server re-computes and verifies match. Hash also goes into the EIP-712 LeaseAgreement payload. Three independent computations of the same hash = catches both upload corruption and tampering.

Regulatory BSA §314

Compliance failure surface

The surface where the most expensive engineering mistakes happen. The Bank Secrecy Act §314 prohibits informing a user that their account is flagged. OFAC, enhanced-KYC, sanctions screening, and jurisdiction restrictions all collapse to the same generic message. The Compliance Officer handles user communication out-of-band.

Watch-outs
  • No retry button. Inviting retry signals "this is the same type of block" — that's a tipping-off signal. Single CTA: "Contact support."
  • All compliance contract reverts collapse to one error class. OfacScreeningFailed, SanctionsListMatch, EnhancedDueDiligenceRequired, JurisdictionRestricted → ofac-restricted. Don't add a variant.
  • Velocity limits are exempted. Velocity is operational rate-limiting, not compliance. You may show the reset time.
  • CO is notified server-side. The Compliance Officer receives the detailed signal via internal channels and handles communication if appropriate. Frontend's job is silence.
  • No Sentry capture with specifics. PII-redacted only. The signal goes to CO via internal channel, not error tracking.
  • This requires CO sign-off. Any change to this surface — copy, layout, CTA — requires Compliance Officer review per CO/GC Pack §6.2.

  • Renewal cadence is hard-coded. 30 days before expiry → banner. 7 days → modal interrupt on next fee-bearing action. Day of expiry → blocks all fee-bearing actions.
  • Wallet rotation is a Phase 1 limitation. Each wallet KYCs independently. Surface this in copy — don't pretend it's not friction.
  • KYC source is abstracted. Default APIKYCSource until AttestationKYCSource decision lands. Consume useKYCStatus(address) only — never the source directly.
  • Tier insufficiency is NOT a compliance failure. KYC-required surfaces the actual reason ("Tier 1 required"). OFAC/sanctions/jurisdiction collapses to the generic compliance surface. Don't conflate.

Architectural decisions baked in · do not invent alternatives
ListingsOff-chain in Phase 2 (PostgreSQL + IPFS for photos). No on-chain footprint until counter-sign. Phase 3 re-evaluates on-chain listing commitments if marketplace scales and bait-and-switch becomes a complaint vector.
SequenceTenant signs first (EIP-712 acceptance becomes the application). Landlord counter-signs to commit. Landlord retains right of refusal. Do not invert.
Pre-approvalTwo-layered: LeaseFi-wide profile (identity + income + credit + history, 60-day validity) AND per-listing criteria match (landlord-set thresholds, computed at view-time). Tenant sees both states. ADR-003 governs vendor stack.
Vendor stackPersona (identity, existing) + Plaid Income (new) + TransUnion SmartMove (new). Backups: Truework + Experian RentBureau. Per ADR-003.
Privacy gatePre-counter-sign: pseudonymous identity + verification badges + criteria-match boolean. Post-counter-sign: full identity revealed. Never expose tenant PII before commit.
Concurrent appsLandlord picks one applicant. When counter-sign succeeds, other applications auto-decline with structured reason codes. Documented trail per FHA.
SettlementSame Tx A (LeaseFactory.create_lease) + Tx B (DepositVault.deposit_funds) as Section 6. Marketplace is entry-point only; the on-chain lease is identical.

Watch-outs · Phase 2 readiness
  • Pre-approval is FCRA-regulated. Credit-based declines trigger FCRA adverse-action notice. Templates are Outside-Counsel-drafted, not engineering-invented. Engineering supplies the delivery channel (email + in-app); CO owns canonical copy. Wrong template = HUD complaint.
  • FHA disparate-impact on criteria thresholds. Outside Counsel reviews allowed landlord-criteria threshold ranges. Some thresholds (e.g., "no evictions ever") have HUD-documented disparate impact on protected classes and are blocked at the listing-criteria UI. Do not let landlords type free-text criteria — structured fields only.
  • Source-of-income discrimination is state-specific. FL/TX/TN Phase 2 launch states do not currently have source-of-income protections. VA Phase 2+ does. Phase 4 states (DC, NY, CA, MA, et al.) do. The accept-Section-8 toggle and "income source" filter are gated by jurisdiction via LandlordCriteriaSource.
  • Privacy gate is load-bearing. Pre-counter-sign tenant detail must never leak full name, exact address, employer, or bank balance. Verification badges and criteria-match booleans only. Test this with an adversarial peer-review pass before launch.
  • Decline-reasoning is mandatory, not optional. Counter-sign is gated on documenting reasons for all non-selected applicants. Skip path does not exist. Audit log persists for fair-housing review.
  • Pre-approval expiry hits at day 60. Renewal flow surfaces at day 53 (banner) and day 58 (modal interrupt on next listing application). Do not let tenants apply with stale pre-approval — landlord sees stale data, landlord makes decisions on stale data, lawsuit risk.
  • Listing edits invalidate prior applications. If a landlord changes rent or term mid-listing, all open applications are invalidated and applicants are notified. Do not silently update behind their back; the EIP-712 acceptance was for the original terms.
  • Backup vendor failover is a real drill, not a hypothetical. ADR-003 §5.4 requires a 48-hour full-failover test before Phase 2 launch. Engineering owns the runbook; CO owns the sign-off.

Scope boundary

What's not in this document

This is the visual spine, not the complete design system. The following surfaces are deliberately deferred — handle them in the order below as Phase 1 progresses.

  • Auth / wallet-connect first impression. Single CTA screen. Easiest Critical 6 flow; build day-one as a warmup.
  • Per-step wizard content for steps 1, 2, 4, 5. Section 6 shows Step 3 in detail and the wizard chrome. Property, tenant, term, and document-upload steps share the same chrome; the form fields differ.
  • Path B fiat on-ramp. Embedded provider widget. Visualize once the on-ramp partner is signed; partner-specific UX constraints.
  • Failed-state UX matrix. Per-error-class copy (insufficient-balance, allowance-insufficient, wallet-locked, network-failure, contract-revert with unknown code). Use the error discrimination table in 05-data-layer-architecture.md §9.
  • Account-switch mid-transaction warning. Three sub-states: aborted-in-signature (clean), aborted-in-pending (chain proceeds), aborted-in-confirmation (track via indexer). Surface the consequence; do not bury.
  • GlobalTxIndicator expanded panel. Multi-transaction list view when ≥2 pending. Click-to-return navigation.
  • Empty states. "No active lease" for tenant home. "No properties yet" for PM dashboard. "No disputes" lists. Template once, reuse everywhere.
  • Loading / skeleton states. Per route. Template once via shadcn Skeleton; do not invent per-component.
  • Settings. KYC tier, wallet management, notification preferences, currency preference. Lower-stakes, higher-volume; default to shadcn settings patterns.
  • Lease detail page. Most-used PM screen. Sections: parties, financial, term, document, activity log, actions (terminate, dispute, request manual deposit release).
  • Notification center. Per-category preferences. Critical security notifications (KYC required, dispute filed) cannot be disabled — only channel selection.
  • Termination & buyout flows. EIP-712 ceremonies. Reuse PreSignatureConfirmation; design the term-specific copy.
  • Mobile responsive variants. The web app must work at mobile widths before the native app ships. Bottom tab bar instead of left sidebar; full-screen forms instead of modals.

All of the above are sized work for the senior frontend engineer. The reference in this document is the irreducible spine — get these nine surfaces right and the rest extends naturally.
